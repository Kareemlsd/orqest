## Executive summary

The Orqest core is well-aligned with its own current scope, but only **one** of the five novel vision features is genuinely shipped (#1 runtime agent design). The other four have varying degrees of foundation:

| Vision feature | Audit verdict |
|---|---|
| **#1 Runtime agent design** | ✅ shipped end-to-end. Polymath sub-agent roster duplicates a latent capability (`MetaOrchestrator._find_or_spawn` already persists specs to memory) — worth consolidating. |
| **#2 Cognitive memory typology** | 🟡 half-shipped. `MemoryEntry.memory_type` is hard-typed `Literal["semantic", "episodic"]` — **procedural is absent**. No kind-aware retrieval strategy. No salience-aware compaction. |
| **#3 Metacognition primitives** | ❌ wide gap. `BaseAgent.run` returns raw `OutputT`. No confidence, no uncertainty, no capability-boundary signal. **Highest-leverage next move.** |
| **#4 Self-healing primitives** | ❌ wide gap. `ToolHook` cannot abort/redirect. `resolve_model` has no fallback cascade. `MCPDiscovery` is a latent self-healing tool with no auto-wiring. No watchdog primitives. |
| **#5 Generative UI** | ❌ wide gap. `ExecutionPlan` is the *closest pattern* in the codebase — but it's hardcoded to plan tasks, not a generic component-spec protocol. |

**Strong prior validated:** the VISION.md prior was *"metacognition is the next move because it gates self-healing."* The audit confirms this — `BaseAgent.run` is the keystone. Without enriched output, `RefinementLoop` cannot use agent self-confidence (its evaluator already accepts a `BaseAgent`, latent), `SubAgentTool` cannot expose confidence in its result, `ContextManager` cannot do salience-driven compaction, and `MetaOrchestrator` cannot re-decompose on confidence drop.

## Top 5 Drift cases

1. **`BaseAgent.run` returns raw `OutputT`** — keystone for metacognition. `agents/base_agent.py:326`.
2. **`ToolHook` methods return `None`** — blocks both metacognition and self-healing. `hooks.py:24-48`.
3. **`MemoryEntry.memory_type` is `Literal["semantic", "episodic"]`** — no procedural. `memory/store.py:31`.
4. **`resolve_model` is single-shot** — no provider fallback cascade. `utils/llm_model.py:62`.
5. **`ExecutionPlan.to_sse_init` is plan-scoped** — generalizable pattern, hardcoded implementation. `plan/execution_plan.py:80-82`.

## Top 5 Latent cases

1. **`MetaOrchestrator._find_or_spawn` persists agent specs to memory** (meta.py:222-295). Polymath's Postgres roster duplicates this — consolidate.
2. **`MetaOrchestrator._spawned_agents` in-memory roster** (meta.py:111). Pair with above.
3. **`RefinementLoop.evaluator` accepts `BaseAgent` instances** (loop.py:51, 156-166). Half of agent-self-evaluation already wired; just needs `EnrichedOutput`.
4. **`MCPDiscovery.search() → DiscoveredServer.to_config() → MCPServerManager.connect() → MCPToolAdapter.adapt`** chain is fully built (discovery.py + client.py + adapter.py). Only missing: the `ToolRegistry`-miss trigger.
5. **`ExecutionPlan` already implements the init+delta UI pattern** — generalizing it is a refactor, not a from-scratch build.

## Top 5 Gap cases

1. **`EnrichedOutput[T]`** — no type wraps `(output, confidence, uncertainty_targets, capability_boundary)`. New `orqest.metacognition` module.
2. **Watchdog subsystem** — `StallDetector`, `LoopDetector`, `RegressionDetector` as EventBus subscribers. New `orqest.healing` module.
3. **Dynamic tool generation** — agent authors → sandbox → register flow. Touches `autonomy/` and a new sandbox protocol.
4. **`UIComponentSpec[T]` registry** — generic component-spec protocol with frontend resolver contract. New `orqest.ui` module.
5. **Orchestration specs** — `PipelineSpec`, `ParallelSpec`, `RouterSpec`, `RefinementLoopSpec` Pydantic models + JSON hydrators. Closes the runtime-agent-design loop (LLM emits topology, not just agents).

## Strategic recommendation

### Critical path (sequenced — A blocks B)

```
[A] EnrichedOutput[T] + MetacognitionHook + BaseAgent.run upgrade   ← KEYSTONE
        │
        ├─ unblocks confidence-driven RefinementLoop convergence
        ├─ unblocks confidence in SubAgentResult
        ├─ unblocks salience-driven ContextManager compaction
        └─ unblocks confidence-drop trigger for MetaOrchestrator re-decomposition
             │
[B]  ToolHook decision protocol upgrade (HookDecision = Continue|Skip|Redirect|Abort)
        │
        ├─ unblocks watchdog-driven recovery actions
        └─ unblocks confidence-aware abort decisions
             │
[C]  Watchdog subsystem (orqest.healing) + resolve_model_with_fallback
        │
        ├─ depends on [A] for regression detection (confidence trend)
        └─ depends on [B] for abort/redirect mechanism
             │
[D]  MCPDiscovery → ToolRegistry auto-wire on capability miss
        │  (small, depends on [B] indirectly — wants to be a hook-driven flow)
             │
[E]  Procedural memory + per-kind retrieval strategy
        │  (parallel track — independent of A/B/C/D)
             │
[F]  Generative UI — UIComponentSpec[T] + component registry + ExecutionPlan refactor
        │  (depends on [A] indirectly: confidence is one of the typed UI events)
```

### Recommended next plan

**Priority A: ship metacognition (`orqest.metacognition` module).**

Concretely:
- New module `orqest/metacognition/` with:
  - `EnrichedOutput[OutputT]` Pydantic model: `output`, `confidence`, `uncertainty_targets: list[str]`, `capability_boundary: bool`
  - `MetacognitionHook` (a `ToolHook`) that runs a self-evaluation prompt after each tool call, emits `metacognition.confidence` events
  - Self-reflection prompt template (loadable via `load_sys_prompt`)
  - `confidence_protocol.py` — pluggable protocol for how confidence is extracted (LLM self-rating, evaluator-based, ensemble)
- Extend `BaseAgent.run_enriched(state) -> EnrichedOutput[OutputT]` (additive — `run` keeps backward compat)
- Wire `RefinementLoop` `use_agent_confidence: bool` flag — uses `EnrichedOutput.confidence` in convergence check
- Wire `SubAgentResult.confidence: float | None`
- Tests + concept doc under `docs/concepts/metacognition.md`

### Audit corrections that surfaced during deep-dive (Plan agents)

The deep-dive (`designs/0[1-3]-*.md`) surfaced six findings the assumption-first audit missed:

1. `RefinementLoop` "latent" claim was overstated — same agent can't rate itself today; only "another `BaseAgent[..., EvalResult]`" works. True self-rating requires `EnrichedOutput`.
2. `MetaOrchestrator._find_or_spawn` already prompts spawned agents for a `confidence` number (`meta.py:267-272`) — latent capability the audit missed. `_extract_confidence` now reads it.
3. "Hooks are fire-and-forget" was framed as semantic; it's an annotation choice. Adding decision-returning protocol is additive, not destructive.
4. `HookRunner` does NOT intercept pydantic-AI's internal tool dispatch. Hooks fire only at compound-flow boundaries (`CompoundTool`, `run_with_retry`, `MetaOrchestrator._execute_subtask`). Skip/Redirect/Abort affect only those flows. Future expansion would need to wrap each `Tool.function` at construction (`as_tool`, `MCPToolAdapter.adapt`).
5. Runtime 5xx during `model.request()` is uncaught by orqest fallback. `resolve_model` only handles resolution-time failure. `FallbackModel` subclasses `pydantic_ai.models.Model` (not wraps) so it intercepts `request()` failures.
6. Polymath frontend has hardcoded SSE event whitelist. Generative UI needs a `GET /event-types` manifest endpoint for frontends to auto-build listener lists. Designed; not yet plumbed (consumer-side follow-on).
