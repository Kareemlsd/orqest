## Goal

Orqest is a **cognitive backbone for autonomous agentic software** ÔÇö not another agent framework, but a substrate others build on.

The animating question: *if I want to build a top-tier multi-agent system, what plumbing should already exist so I can stand it up in a day?* Orqest's reason for being is to be the answer.

### Positioning vs. the 2026 landscape

Every framework picks one stance and locks you into it:

| Framework | Stance | Limitation |
|---|---|---|
| LangGraph | graph-based orchestration, low-level | batteries not included; static topology |
| CrewAI | role-based crews | opinionated; hard to customize |
| AutoGen | conversational multi-agent | research-grade; fragile in production |
| Devin / Manus | vertically integrated products | closed; not composable |
| OpenAI Agents SDK | Assistants API + handoffs | provider-locked |
| Pydantic-AI | typed single agents | no orchestration story |

Orqest aims at a **fourth category: agents as a runtime system** ÔÇö kernel-style cognitive substrate, not graph or crew or product. The LLM designs the topology at runtime; memory is structured by cognitive *kind*; the agent reports its own uncertainty; failure recovery is a primitive; the UI surface is generated by the agent for the task.

### Foundational discipline ÔÇö the litmus test

> "Core Orqest manages the **shape and flow** of intelligence; extensions manage the **matter and action** of the domain."

Litmus test for any feature: *Could a developer building a headless coding assistant use this without knowing what numatics-ai is?* If no, it belongs in a consumer, not Orqest core.

This is the discipline that keeps Orqest neutral enough to power a coding assistant, an autonomous OS, or a scientific-discovery platform from the same primitives.

### Architecture inspirations

- **Claude Code's nO loop** ÔÇö flat agent loop, 3-tier memory, context compaction, worker threads
- **Human cognition** ÔÇö sensory Ôćĺ working Ôćĺ STM Ôćĺ LTM with episodic / semantic / procedural subdivision
- **State-of-the-art frameworks** ÔÇö what they got right (LangGraph's typing, Pydantic-AI's tools) and what they got wrong (rigid topology, scattered observability)
- **Pydantic-AI as substrate** ÔÇö Orqest *consumes* pydantic-ai's typed agents and tools; we add the cognitive layer on top, never re-implement what's below. As pydantic-ai improves, Orqest gets the upgrade for free.

## The five novel features

These are the features that, if shipped well, define the category. Numbered in order of how distinctive they are vs. the existing landscape.

### 1. Runtime agent design ÔÇö *agents that design agents*

The LLM emits a JSON `AgentSpec`; the `AgentFactory` hydrates it into a live `DynamicAgent`; the `ToolRegistry` resolves tools by name; the `MetaOrchestrator` decomposes goals and assigns sub-agents at runtime.

**Why novel:** LangGraph, CrewAI, AutoGen all require declared topology upfront. Orqest treats agent topology as runtime state. The orchestrator can grow its own organization as it discovers what it needs ÔÇö and persist named sub-agents across turns so the roster survives.

**Status:** Ôťů shipped (Phase 3 of the Orqest roadmap). Polymath demonstrates this end-to-end via persistent sub-agent roster. **2026-04-25 update:** `MetaOrchestrator._find_or_spawn` now dual-writes episodic + procedural memory entries; recall is procedural-first with episodic fallback.

### 2. Cognitive memory typology ÔÇö *memory that mirrors human cognition*

Three first-class memory kinds:

- **Episodic** ÔÇö what happened (sessions, traces, prior runs)
- **Semantic** ÔÇö what's true (facts, summaries, learned content)
- **Procedural** ÔÇö how to do things (skills, recipes, learned tool sequences)

**Why novel:** Most frameworks treat memory as "vector store + metadata." Letta gets close but doesn't separate procedural from declarative. The cognitive-science taxonomy lets agents reason about *what they remember* vs *what they did* vs *how they do things* ÔÇö distinct retrieval and storage strategies for each. Long-running agents need this distinction to avoid retrieving stale facts when a procedure is needed.

**Status:** Ôťů shipped (Wave 1.2, 2026-04-25). `MemoryEntry.memory_type` is `Literal["semantic","episodic","procedural"]`. `Skill` / `ToolCallSpec` / `SkillExample` Pydantic shapes carry procedural payload in `structured_content`. Per-kind retrieval via `RetrievalStrategy` Protocol (`SemanticStrategy` / `EpisodicStrategy` / `ProceduralStrategy` with optional injected fuzzy judge). `MemoryConfig.PerKindConfig` carries TTL / decay / version-on-edit policies. Supabase/pgvector backend remains a future track.

### 3. Metacognition primitives ÔÇö *agents that know what they don't know*

Every agent output carries:

- a **confidence score** the agent computed itself
- a **what-I'm-uncertain-about** introspection (which assumption is the bottleneck?)
- a **capability-boundary signal** ("this is outside what I can verify")
- **self-reflection hooks** that fire after every turn

**Why novel:** No mainstream framework treats metacognition as a core primitive. DSPy optimizes prompts but doesn't introspect at runtime. Letta has reflection but it's a memory write, not a confidence-aware output protocol. The difference between an agent that says *"the answer is 42"* and one that says *"42, confidence 0.6, the bottleneck is whether assumption X holds"* is the difference between an agent you deploy and one you babysit. **This is the most distinctive feature on the list.**

**Status:** Ôťů shipped (Wave 1.3, 2026-04-25). `EnrichedOutput[OutputT]` (output + confidence + uncertainty_targets + capability_boundary + protocol_name + metadata). Three concrete `ConfidenceProtocol` strategies ÔÇö zero-cost `StructuredOutputProtocol` (default), +1-call `LLMSelfRatingProtocol`, +k-call `EnsembleProtocol`. `MetacognitionHook` emits `metacognition.confidence` events. Integrated into `BaseAgent.run_enriched`, `RefinementLoop` (`confidence_threshold`/`agent_self_eval`), `SubAgentTool.use_enriched`, `ContextManager.salience_fn` (`confidence_salience` rescues high-salience old messages), `MetaOrchestrator` re-decomposition (reads the latent `confidence` field `_find_or_spawn` already prompts spawned agents to emit).

### 4. Self-healing as primitives, not patterns

- **Watchdogs** ÔÇö first-class subsystems detecting stall, loop, regression
- **Dynamic tool generation** ÔÇö when an agent needs a missing capability, generate the tool at runtime (sandboxed)
- **Model auto-switching** ÔÇö graceful provider fallback (OpenAI Ôćĺ Anthropic Ôćĺ local) without restart

**Why novel:** Manus has rough self-recovery but not as composable primitives. Most framework error handling is "retry the LLM call." Orqest's intent is to make watchdogs and recovery as routine as the EventBus is now ÔÇö wire once, every agent inherits robustness. Combined with metacognition, this is what makes Orqest agents survivable in production without a human pager.

**Status:** Ôťů shipped (Wave 2, 2026-04-25). `HookDecision` discriminated union (Continue/Skip/Redirect/Abort) lifts the hook protocol from observe-only to decision-issuing while staying back-compat (`None` returns auto-wrap to `Continue`). `Watchdog` Protocol + `StallDetector` / `LoopDetector` / `RegressionDetector` (the latter consumes `metacognition.confidence` events). `RecoveryAction` union + `WatchdogHook` translate detection into a `HookDecision`. `FallbackModel` subclasses `pydantic_ai.models.Model` (sticky failover, transient classifier). `resolve_model_with_fallback` accepts a chain. `HealingRunner` is the async-context-manager lifecycle owner. MCP auto-wire (`ToolRegistry.get_or_discover` + `DiscoveryHook` + `PermissionGate`, default `DenyAll`).

### 5. Generative UI ÔÇö *agents that design their own surface*

The agent emits component specs (chart, table, form, plan, takeover dialog) and the frontend hot-loads them via a server-driven UI protocol.

**Why novel:** Vercel's AI SDK has tool-call-rendered components but they're hand-coded per tool. Anthropic's artifacts system is closed. Orqest's vision: **the agent describing the UI surface for the work it's doing**, with a component registry the frontend resolves dynamically. An agent doing scientific discovery emits a `MoleculeViewer` spec; an agent doing finance emits a `RiskHeatmap`. The frontend is a platform, not a product.

**Status:** Ôťů shipped (Wave 3, 2026-04-25). `UIComponentSpec[T]` (generic Pydantic + `component_type` `Literal` discriminator) + `UIDeltaEvent` (replace/merge/append/remove ops with dot-path). `ComponentRegistry` per-Workbench (no module singleton); `default_registry()` auto-loads first-party. `UIEmitter(bus)` for init/delta/remove. Five first-party components: `PlanComponent` / `ChartComponent` / `TableComponent` / `FormComponent` / `TakeoverDialogComponent`. `ExecutionPlan.enable_ui_events()` flag-gated dual emission of `ui.plan.*` events. SSE event-type conventions: `ui.<component_type>.{init,delta,remove}` (helpers in `orqest.ui.events`). Frontend resolver protocol designed; Polymath migration to typed components is consumer-side follow-on.

### Honest read (original)

**What we've proven:** the *batteries-included* claim. A developer wanting to build a Manus-class agent can read Polymath and Orqest as a recipe. Eight days from blank repo to two-pane shell with working autonomous research Ôćĺ benchmark Ôćĺ PDF.

**What we haven't proven yet:** Orqest is *trustworthy* (needs metacognition), *survivable* (needs self-healing), or *a substrate* (needs generative UI). All three are achievable on the current foundation ÔÇö the orchestration, observability, hooks, and event bus are the right primitives to build them on.

**The strategic move from here:** ship Polymath Phase 6 (the demo polish ÔÇö README, video, CHANGELOG, CI) so Orqest has a flagship. Then **prioritize metacognition (Phase 3)** as the next core battery, because it has the highest distance-from-existing-frameworks and the highest leverage for trust. Self-healing builds on metacognition (you can't recover from what you can't detect). Generative UI is the endgame, after the cognitive substrate is solid.

### What this changes for new feature proposals

The litmus test still applies. But the strategic-priority ordering ("close gaps in shipped novel features Ôćĺ ship next missing novel feature Ôćĺ polish") shifts: **all five are shipped**, so the next strategic moves are:

1. **Production memory backend** ÔÇö Supabase pgvector (additive completion of feature #2's deployment surface).
2. **Polymath consolidation** ÔÇö migrate Polymath onto `orqest.ui` (typed component specs) and `orqest.healing` (watchdog wiring); consolidate Polymath's Postgres sub-agent table onto `MetaOrchestrator._find_or_spawn`'s now-procedural persistence.
3. **Concept docs** ÔÇö three new modules need `docs/concepts/{metacognition,healing,generative_ui}.md`.
4. **`ToolSandbox`** ÔÇö generated-tool-code safety surface (Phase 3's deferred safety item; relevant for agents that author + run their own tools, the natural extension of `DiscoverAndRetry` recovery action).
5. **Resilience polish** ÔÇö `request_stream` mid-stream failure handling, watchdog poll-vs-push mode, `FallbackModel._idx` reset-on-success.

A *sixth* novel feature (e.g., direct agent-to-agent messaging beyond `MetaOrchestrator`'s spawn pattern, or runtime orchestration-spec hydration so the LLM emits `PipelineSpec`/`ParallelSpec` JSON) would warrant updating this document with a new revision.

## How to use this document

- **When proposing a new feature:** check it lands inside one of the five novel features, OR clearly belongs in a consumer (per the litmus test). If it's neither, it probably doesn't belong in Orqest core.
- **When prioritizing the roadmap:** the order is (1) close visible gaps in already-shipped novel features, (2) ship the next missing novel feature, (3) polish. Distance-from-existing-frameworks beats incremental polish.
- **When measuring progress:** revisit this document quarterly. Mark the date of the new assessment. Compare against this baseline.
